# Варианты Chrome и Chromium-Gost

## Основной принцип

В репозитории хранится один общий исходный код и две небольшие конфигурации. Полные копии расширения и долгоживущие Git-ветки для браузеров не используются: это предотвращает расхождение исправлений.

Источники истины:

- `extension/` — общее поведение, интерфейс, адаптеры и тестируемые контракты;
- `extension/manifest.base.json` — общие разрешения и точки входа Manifest V3;
- `variants/chrome/config.json` — версия и политика Chrome;
- `variants/chromium-gost/config.json` — версия и политика Chromium-Gost;
- `variants/*/INSTALL.txt` — инструкция, которая попадает только в соответствующий ZIP.

`build/` и `dist/` — производные каталоги. Их содержимое можно удалить и полностью восстановить командами сборки; вручную оно не редактируется и в Git не коммитится.

## Матрица различий

| Параметр | Chrome | Chromium-Gost |
|---|---:|---:|
| Отображаемая версия | `0.8.4-chrome` | `0.8.4-gost` |
| Минимальная версия браузера | 120 | 143 |
| Ожидание готовности команды | 0 мс | 1500 мс |
| Пауза между актами | 0 мс | 5000 мс |
| Поздняя проверка после таймаута | 0 мс | 2500 мс |
| Максимум попыток нативного старта | 1 | 2 |
| Окно сопоставления загрузки | 35000 мс | 40000 мс |

Все значения задаются в `nativeDownloads` соответствующего `config.json` и попадают в генерируемый `shared/variant-config.js`.

## Где редактировать

- Общая функция или исправление для обоих браузеров: редактировать `extension/`.
- Только тайминги, название, версия или минимальный браузер: редактировать нужный `variants/*/config.json`.
- Существенно различающееся поведение: сначала оформить параметром или отдельной стратегией в общем коде; не размножать проверки браузера по разным файлам.
- Пользовательская памятка конкретной сборки: редактировать `variants/*/INSTALL.txt`.
- Не редактировать `build/*/manifest.json`, `shared/variant-config.js`, `BUILD-INFO.json` и ZIP: следующая сборка перезапишет их.

## Версионирование

Номера вариантов независимы. Числовой `manifest.version` соблюдает формат Chromium, а `manifest.versionName` добавляет канал:

- Chrome: `0.8.4` и `0.8.4-chrome`;
- Chromium-Gost: `0.8.4` и `0.8.4-gost`.

Если изменена только политика Gost, повышается только Gost. Если общий код меняет поведение обеих сборок, версии обеих конфигураций должны быть пересмотрены отдельно. Имя ZIP всегда включает вариант и числовую версию.

Рекомендуемые Git-теги после успешного выпуска:

```text
chrome-v0.8.4
chromium-gost-v0.8.4
```

Тег ставится только после проверки реального браузера и должен указывать на коммит, из которого создан переданный ZIP.

## Сборка и проверка

```bash
npm run build
npm run check
npm run package
```

Контроль состоит из следующих уровней:

1. Конфигурация проверяется на обязательные поля, числовую версию, имя архива и согласованность временных окон.
2. Для каждого варианта создаётся отдельный `manifest.json` и проверяются все ссылки на runtime-файлы.
3. Unit-тесты подтверждают различия политик Chrome и Chromium-Gost.
4. Smoke-тест загружает обе распакованные сборки в реальный Chromium и сверяет название, версию, popup и service worker.
5. Упаковщик проверяет целостность обоих ZIP и создаёт `release-index.json` с SHA-256, а также `SHA256SUMS`.
6. CI выполняет тот же набор и публикует обе сборки одним контролируемым artifact.

## Выпуск и откат

Перед передачей сравните имя ZIP, `versionName` в `release-index.json` и SHA-256. Предыдущие ZIP храните неизменными по версиям.

На рабочем компьютере браузер должен загружать постоянную папку активного варианта. При обновлении заменяется её содержимое, затем нажимается «Обновить» в `chrome://extensions`. Для отката содержимое заменяется файлами предыдущего сохранённого ZIP.

Две сборки можно хранить рядом, но нельзя одновременно включать в одном профиле: обе подписываются на назначение имени загрузки и будут конфликтовать.
